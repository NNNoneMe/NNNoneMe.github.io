秒杀系统业务分析

系统业务流程如下

从图中可以发现，整个系统就是围绕库存做的系统。用户秒杀商品成功，对于系统的操作而言就是：1.减库存。2.记录用户的购买明细。

下图是用户对库存的业务分析：

记录用户的秒杀成功信息，简化系统需求，我们只需要记录：1.谁秒杀成功了。2.秒杀成功的时间。3.秒杀的商品。这三者组成了用户秒杀成功的信息，也就是用户的购买明细。

业务分析中我们使用了事务来控制“减库存”和“记录购买明细”这两步操作。那么，我们为什么要使用事务？

看如下这两者情况：

1.用户秒杀成功，我们记录了购买明细，却没有减库存。导致商品超卖。

2.减少了库存，却没有用户的购买明细。导致商品少卖。

对于以上两种情况，若是没有事务的支持，我们的系统就是存在故障，在实际生产当中，也会造成很大的经济损失。在MySQL中，它内置的事务机制，可以准确的帮我们完成减库存和记录用户购买明细的过程，避免上述两种情况的发生。

MySQL实现秒杀系统难点分析

当用户A秒杀id为10的商品时，此时MySQL需要进行的操作是:

1.开启事务

2.更新商品的库存信息

3.添加用户的购买明细，包括用户秒杀的商品id以及唯一标识用户身份的信息（系统中简化为用户电话）

4.提交事务

若此时有另一个用户B也在秒杀这件id为10的商品，他就需要等待，等待到用户A成功秒杀到这件商品然后MySQL成功的提交了事务他才能拿到这个id为10的商品的锁从而进行秒杀，而同一时间是不可能只有用户B在等待，肯定是有很多很多的用户都在等待拿到这个行级锁。 秒杀的难点就在这里，如何高效的处理这些竞争？如何高效的完成事务? 这一部分会在后面的处理高并发的优化中去记录。

秒杀系统API之Service层开发

开发前的思考

在开始Service层编码前，我们会发现我在实现Dao层的时候，只完成了针对表的相关操作，包括编写接口方法和映射文件中的sql语句，并没有写逻辑代码，例如对多个Dao层方法的拼接，当用户秒杀成功后，我们需要减少库存（调用SeckillDao接口）和增加用户的购买明细（调用SuccessKilledDao接口）。像这样涉及到业务逻辑的代码，我们需要在Service层完成。很多初学者在开发项目的过程中，喜欢把Dao层的代码写得很复杂，恨不得把业务逻辑的代码加入其中，却忘了Dao层就是做数据访问，只需要进行数据的访问操作，至于如何去拼接这些操作，交给Service层去实现。下面正式开始Service层编码。

Service接口设计思路

如何站在“使用者”的角度设计接口方法？

这里假如我就是这个“秒杀系统”使用者，我要秒杀一些商品，我希望这个系统能怎样做？

  • a.至少得提供能秒杀商品的列表吧，然后告诉我什么时候可以开始秒杀，库存还剩多少等信息；
  • b.当我选中某件商品时，我能看的该商品是否正在进行秒杀；
  • c.如果还未到秒杀时间，要是能给我提示该多好
  • d.到了秒杀时间，我点击秒杀按钮就可以完成秒杀

以上就是我作为使用者的一个需求，当然不能完全作为一个使用者去想这些步骤，因为你得考虑到这样的需求是否能够实现。针对以上需求，在Service实现类中，我们可以设计一个获取全部商品方法：getSeckillList()和获取具体商品的方法：getbyId() ； 然后设计一个秒杀接口暴露方法：exportSeckillUrl()和执行秒杀的方法：excuteSeckill()；设计这4个方法，基本满足需求。

为什么使用dto封装返回数据？

在Service方法设计中，我们提前设计了几个dto类型，用以封装接口方法返回的数据，有点小伙伴看到这里或许有一点迷惑，为什么要这样做？

其实，这就是java面向对象思想的体现。试想，如果不进行封装返回数据，我们的接口方法中需要返回什么类型的值？举个例子吧：就拿exportSeckillUrl()这个方法来说。此方法需要给用户暴露我们的接口地址，你可能回想到，为什么不直接返回接口地址呢？获取不到的时候返回空嘛。但是，在实际编程中，程序所需要的信息往往不可能只有一个，还是上面的方法，当我们能够秒杀的时候，返回地址就行了，但是当我们不能秒杀的时候，程序往往需要跟多信息，比如能够秒杀该商品的时间等。所以，我们将程序可能需要的信息进行封装，通过不同的构造方法来返回不同的数据。不管秒杀是否成功，我们只需要返回的对象就行了，而不必去在意细节。
